因为Git是分布式版本控制系统，所以，每个机器都必须自报家门：你的名字和Email地址，这样远程仓库才知道哪次提交是由谁完成的。

## git的一些知识

![image-20250725125616717](/home/rm_young/.config/Typora/typora-user-images/image-20250725125616717.png)

git add 可以把之前我们初始化仓库时，并没有这个文件，但后面我因为需要，所以我新创建了一个文件，就比如我的`.gitignore`这个文件，**这个文件之前是没有进行跟踪的**，可以把这个文件添加到**暂存区**中，并进行跟踪，这样后面就可以做同步更改了。

**管理修改**：git只会提交(commit))暂存区的内容

**撤销修改**:

1.在工作区对文件进行了修改，但**未添加到暂存区**，可以用一下指令撤销修改

```c++
git checkout -- 文件名
```

2.对文件做了修改，并且**将修改提交到了暂存区**，想还原回原来的文件，分两步走

```c++
git reset HEAD file   //回到场景1
//接着就按场景一操作就好了
```

3.修改并提交到暂存区，并提交到仓库，用版本回退就好了。

**对比文件不同**：

```//
git diff HEAD -- 文件名  //上一个版本与当前版本对比
git diff HEAD HEAD^ -- 文件名 //上上个版本与上个版本的对比
```

**删除文件**:

删除的是我们在**工作区**的文件

```c
rm 文件名
git rm 文件名 //把删除的改动添加到暂存区
git commit -m '版本名称' //版本名称是自主命名的
```

这个删除文件==本质也是创建版本==，在上面的各个阶段，我们也可以合理运用上**撤销修改**,比如运行完指令`rm 文件名`,后悔了，可以用`git checkout -- 文件名`,再比如**已经把东西提交到仓库了**，但我们后悔文件删除了，我们就可以使用版本回退。

### 管理策略

git分支图的指令，可以很形象看到分支的对应关系

```c++
git log --graph --pretty=online
```

2个点

1. 当我们创建了一个新分支，并且在两个分支（一个是原有的分支）分别进行了提交，且没有冲突（没有修改同一个文件），这时，我们合并分支时，git会帮你提交一次，所以会弹出==一个界面==，在这个界面里，让你输入此次提交的名称，然后用ctrl+s保存，输入Y，enter退出。

2. 当我们创建了一个新分支dev，然后在新分支上进行了提交，然后用`git checkout <name> `切换回原来的分支，但相比==1的情况==原分支是没有进行提交的，此时如果直接用`git merge dev`的话，使用的是快速合并模式。 但如果我们想要**强制禁用快速合并**呢？可以使用下面的指令 

   ```c++
   git merge --no-ff -m '版本名称'
   ```

### bug分支修改

这个功能使用于我们有临时任务需要完成，但是主线任务还没完成的情况下

情景是这样的，我们现在有两个分支，==一个是master分支，一个是dev分支==，现在有临时bug，为了修复bug，我们可以创建一个分支来进行修复，接着把这个分支合并到master分支里，但注意这里不建议用快速合并，==因为会使这个分支直接消失没有记录，导致我们后续难以审查这里进行了什么修改==，而不是你之前快速合并会把三个分支合并到一个分支的理解。

**强制禁用快速合并的优点**：

1. 清晰的历史记录： 明确标记出这是一个为了修复 bug 而进行的合并操作，即使修复分支本身只有一两个提交。这使历史更容易理解。

2. 易于定位： 如果将来发现这个修复引入了新问题，你可以很容易地找到那个合并提交，查看它的修改内容或进行回滚。
3. 保留上下文： 记录了“修复分支曾存在”这一事实（即使分支后来被删除）。所以得用强制禁用快速合并的指令 git merge --no-ff -m '版本'

下面是流程

```go
git stash //这里保存dev目录当前工作现场。便于后续恢复
git checkout master //替换至主分支
git checkout -b bug-001 //创建新分支，并切换
//  修复bug
git add 修改的文件名
git commit -m 修改的文件名
git checkout master //切换分支
git merge --no-ff -m '修复bug' bug-001
git branch -d bug-001
git stash list //列出保存线程
git stash pop //恢复工作现场
```

### Github推送代码

我们一般从开发中，都是新建一个分支来进行开发，开发完成后，代码进行提交(`git commit -m `)后，还得==添加远程仓库==，可以用推送代码的指令，`git push 远程分支 本地分支 `。

还可以跟踪我们的本地提交和远程提交的区别，怎么说这个区别呢？就是你在本地执行完`git commit -m `后，代码还没有推送的情况下，此时和远程提交就有了区别。我们可以利用下面的指令**跟踪这种区别**.

```python
git branch --set-upstream-to=origin/master master #master是自主确定的
```

从远程分支上拉取代码

```python
git pull origin 分支
```

### 同步更新Github

五、Git分支管理
Git分支说明
- main - 项目主分支，存储可以稳定使用的代码
- dev - 项目开发分支，存储已经开发好的代码
- feature-* - 特性分支，存储正在开发的一个功能
- release-* - 对比赛中使用过的代码进行存档
工作流程
- 对于每个要开发的功能新建一个feature-*分支，在上面进行开发
- 每个分支的功能做完之后合并到dev分支，并且删除对应的feature-*分支
- dev分支的代码可以稳定运行后将代码合并到main分支
注意事项
- 多人协同开发时不建议直接在main分支和dev分支上进行开发
